# ChurnSense

A production-grade customer churn prediction system built for a telecom company. Predicts which customers will churn, explains why, forecasts when, and generates personalized retention strategies using an LLM pipeline.

**Live Demo:** https://churnsense-app.streamlit.app
## The Problem

Telecom companies lose 15–25% of their customer base every year to churn. Acquiring a new customer costs 5–7x more than retaining an existing one. Most churn solutions stop at a prediction score. ChurnSense goes further — it tells the retention team not just who is leaving, but why they are leaving, how long they have left, and exactly what offer to make them.

## What This System Does

**Predicts** churn probability for every customer using an XGBoost model trained with Optuna hyperparameter optimization (AUC-ROC: 0.846).

**Forecasts** survival probability at 30, 60, and 90 days using a Cox Proportional Hazards model — so the retention team knows how urgently to act on each customer.

**Explains** every prediction using SHAP values, producing a ranked list of the top risk factors driving that specific customer toward churn.

**Segments** at-risk customers into behavioral groups by clustering on SHAP values rather than raw features — so segments are defined by why customers churn, not just what they look like.

**Generates** a personalized 3-action retention plan for each customer using GPT-4o-mini, incorporating their risk factors, contract type, tenure, and segment profile.

**Serves** everything through a REST API with three endpoints and an interactive Streamlit dashboard built for retention teams.

## API Reference

### POST /predict

Returns churn score, risk tier, and survival probabilities at 30, 60, and 90 days.

```json
{
  "churn_score": 0.8691,
  "churn_percent": "86.9%",
  "risk_tier": "critical",
  "survival_day_30": 0.04,
  "survival_day_60": 0.01,
  "survival_day_90": 0.0,
  "churn_prob_day_30": 0.96,
  "churn_prob_day_60": 0.99,
  "churn_prob_day_90": 1.0
}
```

### POST /explain

Returns the top 5 SHAP risk factors driving this customer's churn probability.

```json
{
  "churn_score": 0.8691,
  "risk_tier": "critical",
  "top_factors": [
    {
      "feature": "is_month_to_month",
      "direction": "increases churn risk",
      "magnitude": 0.4358,
      "value": 1.0
    }
  ]
}
```

### POST /retention-plan

Returns a personalized 3-action retention strategy generated by GPT-4o-mini.

```json
{
  "segment": "early-lifecycle",
  "urgency": "critical",
  "summary": "New fiber customer on month-to-month with no established loyalty.",
  "actions": [
    {
      "type": "discount",
      "title": "Annual plan discount",
      "message": "Switch to an annual plan and save 15% every month.",
      "offer": "15% discount for 12 months on annual contract",
      "expected_impact": "high"
    }
  ]
}
```

## Key Technical Decisions

**Why survival analysis?**
Most churn projects predict a binary yes/no. Survival analysis predicts when — giving the business a time-bounded probability that changes the urgency of outreach.

**Why SHAP-based segmentation?**
Clustering on raw features groups customers by what they look like. Clustering on SHAP values groups customers by why they churn — producing actionable segments like price-sensitive or service-dissatisfied.

**Why a fallback retention engine?**
The LLM engine includes a rule-based fallback that activates when the API is unavailable or rate-limited. The system always returns a retention plan.

**Why business metrics alongside ML metrics?**
AUC-ROC does not tell a retention manager whether to run a campaign. The evaluate module finds the threshold that maximizes profit — not the one that maximizes F1.
