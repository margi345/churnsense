**Requirements:** Python 3.10+, Git

**1. Clone the repository**

```bash

cd churnsense
```

## API Reference

All endpoints accept a JSON body with the customer's raw feature values.

### POST /predict

Returns churn score, risk tier, and survival probabilities at 30, 60, and 90 days.

```json
{
  "churn_score": 0.8691,
  "churn_percent": "86.9%",
  "risk_tier": "critical",
  "survival_day_30": 0.04,
  "survival_day_60": 0.01,
  "survival_day_90": 0.0,
  "churn_prob_day_30": 0.96,
  "churn_prob_day_60": 0.99,
  "churn_prob_day_90": 1.0
}
```

### POST /explain

Returns the top 5 SHAP risk factors driving this customer's churn probability.

```json
{
  "churn_score": 0.8691,
  "risk_tier": "critical",
  "top_factors": [
    {
      "feature": "is_month_to_month",
      "direction": "increases churn risk",
      "magnitude": 0.4358,
      "value": 1.0
    }
  ]
}
```

### POST /retention-plan

Returns a personalized 3-action retention strategy generated by GPT-4o-mini.

```json
{
  "segment": "early-lifecycle",
  "urgency": "critical",
  "summary": "New fiber customer on month-to-month with no established loyalty.",
  "actions": [
    {
      "type": "discount",
      "title": "Annual plan discount",
      "message": "We appreciate you being with us. Switch to an annual plan and save 15% every month.",
      "offer": "15% discount for 12 months on annual contract",
      "expected_impact": "high"
    }
  ]
}
```

## Key Technical Decisions

**Why survival analysis?**
Most churn projects predict a binary yes/no. Survival analysis predicts when — giving the business a time-bounded probability that changes the urgency of outreach. A customer with 90% churn probability in 30 days needs a different response than one with 90% probability over 12 months.

**Why SHAP-based segmentation?**
Clustering on raw features groups customers by what they look like. Clustering on SHAP values groups customers by why they churn — which produces actionable segments like price-sensitive or service-dissatisfied rather than meaningless demographic clusters.

**Why a fallback retention engine?**
The LLM engine includes a rule-based fallback that activates when the API is unavailable or rate-limited. This means the system always returns a retention plan — a hard requirement for any production deployment.

**Why business metrics alongside ML metrics?**
AUC-ROC does not tell a retention manager whether to run a campaign. The evaluate module computes revenue saved, campaign cost, and net business value at every threshold, then finds the threshold that maximizes profit — not the one that maximizes F1.

## Dataset

IBM Watson Telco Customer Churn — 7,043 customers, 21 features. Available on Kaggle at `blastchar/telco-customer-churn`.

The dataset contains one known data quality issue: the `TotalCharges` column stores numeric values as strings with whitespace for new customers. This is handled explicitly in `src/data/cleaner.py`.
